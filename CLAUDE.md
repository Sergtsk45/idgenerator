## Роль и стиль работы

- Ты помощник в разработке ПО. Пиши качественный код и документируй процесс разработки.
- Перед каждым значительным шагом спрашивай подтверждения у пользователя.
- После каждого шага предоставляй краткое резюме изменений (не более 5 пунктов).
- При возникновении технических проблем предлагай 2–3 альтернативных подхода.
- Если задача слишком объёмная — предлагай разбить её на подзадачи.
- Сохраняй контекст текущей задачи и общую цель проекта.
- Следуй архитектурным решениям из `docs/project.md`.

  ```markdown
  ### Добавлено
  - ...
  ### Изменено
  - ...
  ### Исправлено
  - ...
  ```

  ```markdown
  ## Задача: [Название]
  - **Статус**: [Не начата / В процессе / Завершена]

  - **Шаги выполнения**:
    - [x] Завершённый шаг
    - [ ] Текущий шаг
  ```

- Компоненты: `ai_docs/develop/components`

## Принципы кода

- **SOLID, KISS, DRY** — обязательно.
- Соблюдай единый стиль кодирования (линтеры, pre-commit hooks).
- Не оставляй неиспользуемый код и комментарии.
- Проводи code review всех изменений.
- Не добавляй `console.log` в production-код.
- Не хардкодь значения, которые должны быть конфигурируемыми.

## Git-workflow

### Именование веток

```
<type>/<short-description>
```

| Тип | Назначение | Пример |
|-----|-----------|--------|
| `feature/` | Новый функционал | `feature/user-auth` |
| `fix/` | Исправление бага | `fix/login-redirect` |
| `hotfix/` | Срочное исправление prod | `hotfix/payment-crash` |
| `refactor/` | Рефакторинг | `refactor/api-layer` |
| `docs/` | Документация | `docs/api-readme` |
| `test/` | Тесты | `test/auth-coverage` |
| `chore/` | Обслуживание | `chore/update-deps` |

- Только строчные буквы, дефисы вместо пробелов.
- `main` — production-ready код; feature-ветки мержатся через PR.

### Коммиты (Conventional Commits)

```
<type>(<scope>): <subject>

[optional body]
```

| Тип | Описание |
|-----|---------|
| `feat` | Новый функционал |
| `fix` | Исправление бага |
| `docs` | Документация |
| `refactor` | Рефакторинг без изменения поведения |
| `test` | Тесты |
| `chore` | Обслуживание/зависимости |
| `perf` | Производительность |
| `style` | Только форматирование |

Правила subject: строчные, без точки в конце, повелительное наклонение, не более 50 символов.

### Pull Request

- Оптимальный размер: < 400 строк, максимум 1000.
- Перед созданием: тесты проходят, ветка обновлена, нет конфликтов.

### Запрещено

- Force push в `main`.
- Коммитить секреты, токены, API-ключи.
- Коммитить большие бинарные файлы (> 95 MB).
- Оставлять WIP-коммиты в PR.

## Безопасность

**НИКОГДА:**
- Не коммить секреты/API-ключи/токены в git.
- Не использовать `eval()` или `Function()` на пользовательском вводе.
- Не хранить пароли в открытом виде (использовать bcrypt/argon2).
- Не конкатенировать SQL-строки (использовать параметризованные запросы).
- Не полагаться только на клиентскую валидацию.

**ВСЕГДА:**
- Валидировать весь пользовательский ввод на сервере.
- Использовать HTTPS в production.
- Добавлять `.env` в `.gitignore`.

## Тестирование

Структура AAA (Arrange / Act / Assert):

```typescript
it('should ...', () => {
  // Arrange
  // Act
  // Assert
});
```

**Тестировать:** публичные API, бизнес-логику, граничные случаи, преобразование данных.

**Не тестировать:** приватные методы, код фреймворков, простые геттеры/сеттеры.

**Мокировать:** внешние сервисы, БД, зависящий от времени код, файловую систему.

## База данных (PostgreSQL + Drizzle)

### Именование

- Таблицы/колонки в БД: `snake_case`.
- Свойства в TS: `camelCase` (Drizzle маппит автоматически).
- Для модулей-интеграций использовать префиксы: `chat_conversations`, `chat_messages`.

### Типы данных

- **PK:** `GENERATED BY DEFAULT AS IDENTITY` (или UUIDv7 при необходимости, не random v4).
- **Внешние ID (Telegram):** по умолчанию `text`.
- **Дробные величины:** `numeric(p,s)` по доменам:
  - `numeric(12,2)` — часы
  - `numeric(14,2)` — площади (м²)
  - `numeric(14,3)` — объёмы (м³), массы (т)
- **Дата:** `date`; **время:** `timestamptz` (UTC).
- **Статусы:** `text + CHECK(status IN (...))` + Zod enum в коде.

### Аудитные поля (обязательны для бизнес-таблиц)

```sql
created_at timestamptz not null default now(),
updated_at timestamptz not null default now()  -- обновляется только триггером
```

`updated_at` **обновлять руками запрещено** — только через триггер `set_updated_at()`.

### JSONB

- **Можно:** сырые AI-данные для отладки, нестабилизированные расширения.
- **Нельзя:** поля для фильтрации/сортировки, ключи связей, агрегируемые значения.

### Ограничения и индексы

- Все FK — с явной политикой `ON DELETE RESTRICT / CASCADE / SET NULL`.
- `CASCADE` — только "вниз по ownership" (дочерняя сущность не живёт без родителя).
- Индексы добавлять вместе с фичей.
- Для статусов — **partial indexes** по "горячим" значениям.
- Soft delete: уникальные индексы делать частичными `WHERE deleted_at IS NULL`.

### Чеклист перед изменением схемы БД

- [ ] Определён источник истины vs производные данные.
- [ ] Выбраны типы данных по правилам.
- [ ] Указаны `NOT NULL`, `DEFAULT`, смысловые `CHECK`, `UNIQUE`.
- [ ] Для каждой связи — FK и политика `ON DELETE`.
- [ ] Добавлены нужные индексы.
- [ ] Рассмотрены partial indexes для статусов.
- [ ] Для внешних источников — idempotency/dedup ограничения.
- [ ] Миграция безопасная: добавление → backfill → ужесточение.
- [ ] Обновлены `shared/schema.ts`, валидаторы Zod.
- [ ] Обновлены `docs/changelog.md` и `docs/tasktracker.md`.

## Пуш на GitHub (чеклист)

1. Проверить статус: `git status`, `git rev-parse --abbrev-ref HEAD`.
2. Проверить большие файлы (порог 95 MB) в рабочем дереве и индексе.
3. Проверить большие blob'ы в истории ветки.
4. Если найдены большие файлы — исключить через `git rm --cached` + `.gitignore`.
5. Стандартная последовательность:
   ```bash
   git fetch origin
   git add <specific-files>
   git commit -m "feat: описание изменений"
   git push -u origin $(git rev-parse --abbrev-ref HEAD)
   ```

## Агенты (роли при сложных задачах)

| Агент | Роль |
|-------|------|
| **planner** | Разбивает задачу на подзадачи, создаёт план в `ai_docs/develop/plans/` |
| **worker** | Реализует код (только код, без документации) |
| **reviewer** | Проверяет качество кода перед коммитом |
| **senior-reviewer** | Проверяет архитектуру и технические решения |
| **refactor** | Рефакторинг без изменения поведения |
| **debugger** | Получает отчёт об ошибках и исправляет их |
| **test-runner** | Запускает тесты и линтер, отчитывается — не исправляет |
| **security-auditor** | Аудит безопасности (auth, API, sensitive data) |
| **documenter** | Обновляет документацию после изменений кода |

### Рабочее пространство оркестрации

Хранится в `.cursor/workspace/active/orch-{timestamp}-{slug}/`:
- `progress.json` — метаданные оркестрации
- `tasks.json` — статус задач
- `links.json` — ссылки на план и отчёт

Завершённые оркестрации перемещаются в `.cursor/workspace/completed/`.
