# TelegramJurnalRabot — перечень улучшений и расширений

Ниже — идеи улучшений, сгруппированные по приоритетам.  
Обозначения: **P0** — критично/блокирует, **P1** — важно, **P2** — полезно.

## P0 — исправления целостности и качества данных
- **Синхронизировать API-контракт и реализацию**
  - В `shared/routes.ts` есть `POST /api/messages/:id/process`; нужно либо реализовать на сервере, либо убрать/переименовать в контракте.
- **Убрать “debug overwrite” при импорте ВОР**
  - Сейчас импорт (через UI) пытается очищать `works` перед загрузкой. В прод-режиме это опасно: нужны режимы “добавить/обновить/перезаписать” с явным подтверждением.
- **Перенести импорт ВОР на сервер**
  - Клиентский парсинг Excel + множество `POST /api/works` плохо масштабируется и нестабилен (таймауты/частичная загрузка). Лучше: `POST /api/works/import` с файлом и серверной валидацией/транзакцией.
- **Гарантировать корректную обработку AI-ответа**
  - Строгая Zod-схема для `normalizedData`, валидация и нормализация (числа, даты, единицы).
- **Надёжная обработка ошибок AI**
  - Очередь/повторная обработка, статусы “pending/processed/failed”, логирование причин.

## P1 — функциональные улучшения продукта (АОСР/журнал)
- **Настоящая интеграция с Telegram**
  - Поддержка Telegram MiniApp контекста (userId из Telegram initData), базовая авторизация/проверка подписи.
  - Альтернатива: бот (Telegraf/grammY) + веб-приложение для просмотра/актов.
- **Голосовые сообщения**
  - Загрузка аудио → ASR (распознавание) → нормализация → сохранение исходного аудио/текста.
- **Редактирование нормализованных данных**
  - UI для ручной правки: workCode, количество, дата, локация + “подтвердить”.
- **Контроль объёмов по ВОР**
  - Сводка “план/факт”, предупреждения о превышениях, прогресс по позициям.
- **Генерация документов**
  - Экспорт акта в PDF/DOCX (шаблоны), выгрузка, история версий.
- **Вложения к акту**
  - UI и API для загрузки/привязки файлов (сертификаты, паспорта, фото), типизация вложений.
- **Поиск/фильтры**
  - По сообщениям/работам/актам: дата, участок, статус, код ВОР.

## P1 — AI/матчинг по ВОР (качество извлечения)
- **Улучшить сопоставление “работа ↔ ВОР”**
  - Синонимы в `works.synonyms` генерировать автоматически (LLM/batch) и хранить нормализованные формы.
  - Эмбеддинги и семантический поиск (pgvector) для выбора workCode.
- **Единицы измерения и конвертация**
  - Нормализатор единиц (м³/м3, т/тонн, м2/м²), явные правила.
- **Контекст по объекту**
  - Справочники: локации/участки/бригадиры/подрядчики; использовать как контекст для извлечения.

## P1 — безопасность и эксплуатация
- **Аутентификация и разграничение доступа**
  - Мульти-пользовательский режим, роли (инженер/прораб/админ), разделение данных по проектам/объектам.
- **Rate limiting и защита API**
  - Ограничения на публичные эндпоинты, защита от спама сообщений и тяжёлых импортов.
- **Аудит-лог**
  - Кто/когда импортировал ВОР, кто создал/сгенерировал акт, кто редактировал нормализацию.

## P2 — UX/качество и DevEx
- **E2E/интеграционные тесты**
  - Критический путь: импорт ВОР → сообщение → нормализация → генерация акта.
- **Наблюдаемость**
  - Корреляционные id, структурные логи, метрики (кол-во обработок/ошибок, latency OpenAI).
- **Фоновые задачи**
  - Вынести AI-обработку в фон (очередь/worker), UI прогресс/статусы.
- **Оптимизация фронта**
  - Виртуализация списков сообщений/работ, кеширование, пагинация.
- **i18n расширение**
  - Полное покрытие RU/EN + форматирование чисел/дат по локали.

## Предлагаемая дорожная карта (очень кратко)
- **Итерация 1 (P0)**: контракт API ↔ реализация, безопасный импорт, валидация normalizedData.
- **Итерация 2 (P1)**: Telegram userId, ручная правка, прогресс по ВОР.
- **Итерация 3 (P1/P2)**: генерация PDF/DOCX, вложения, фоновые задачи и наблюдаемость.
